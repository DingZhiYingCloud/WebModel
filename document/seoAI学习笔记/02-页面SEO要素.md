# 页面 SEO 要素 — 标题/描述/URL/图片

> 来源：Google Search Central 官方文档
> 学习日期：2026-04-27

---

## 一、标题链接 (Title Link)

标题链接是搜索结果中**用户决定是否点击的主要依据**。

### Google 生成标题链接的信息源（按优先级）

| 优先级 | 来源 |
|--------|------|
| 1 | `<title>` 元素中的内容 |
| 2 | 网页上显示的主要视觉标题 |
| 3 | `<h1>` 元素 |
| 4 | `og:title` meta 标记 |
| 5 | 经过样式处理后更大更醒目的内容 |
| 6 | 网页中所含的其他文字 |
| 7-9 | 锚文字、指向链接、WebSite 结构化数据 |

### 最佳实践

| 应该做 ✅ | 不应该做 ❌ |
|-----------|------------|
| 每个网页指定**独立**标题 | 关键字堆砌（如"Foo, foo bar, foobars"） |
| 编写**简练描述性**文字 | 重复或样板化文字 |
| 明确主要标题（大字体、`<h1>`） | 不必要的过长文字 |
| 简明宣传品牌（如"网站名：文章标题"） | 在标题中添加价格等频繁变化信息 |
| 语言与页面内容一致 | 在所有页面使用相同长标题 |

### 常见问题

- **不完整的 `<title>`**：如 `| 网站名` — Google 会自动补充
- **过时的 `<title>`**：日期未更新 — Google 会用页面可见标题
- **不准确的 `<title>`**：与内容不符 — Google 会修改
- **样板文字**：多个页面标题雷同 — Google 会插入区分信息

> 更改后需几天到几周才在搜索结果中生效。

## 二、元描述 (Meta Description)

### 生成机制

- 摘要主要根据**网页内容本身**创建
- meta description 是**次要来源**（如果比网页内容更准确才使用）
- **同一页面，不同搜索词可能展示不同摘要**

### 最佳实践

| 原则 | 说明 | 示例 |
|------|------|------|
| **每页独特** | 避免所有页面使用相同描述 | 首页用网站级描述，文章用文章级描述 |
| **包含关键信息** | 不必是完整句子，可含结构化信息 | "作者：X；日期：2026-04-27；主题：SEO" |
| **程序化生成** | 大型网站可自动生成，但应简单易懂 | 从数据库字段组合描述 |
| **高质量** | 具体、详细、概括整个网页 | 见下方对比 |

### 高质量 vs 低质量描述对比

| 效果欠佳 ❌ | 效果较好 ✅ |
|------------|-----------|
| 缝纫用品、纱线、彩色铅笔...（关键字列表） | 汇集缝纫衣物所需的各类用品。营业时间周一至周五8-5。（具体信息） |
| 本地新闻，送货上门。（每篇新闻相同） | 当地老人在活动前夜偷走了礼物…（特定新闻摘要） |
| 鸡蛋是欢乐源泉…（未概括网页） | 完整指南：1小时内掌握各类鸡蛋烹饪方法（概括网页） |
| 自动铅笔（太短） | 自动削尖的铅笔，含可补充2B铅芯。50支以上免运费。（具体详细） |

### 控制摘要显示

- `nosnippet` — 完全阻止摘要
- `max-snippet:[number]` — 限制长度
- `data-nosnippet` — 阻止页面特定部分

> meta description 没有长度限制，但搜索结果中会根据设备宽度截断。

## 三、URL 结构

### 核心原则：简洁、描述性、以人为本

| 最佳实践 ✅ | 不推荐 ❌ |
|------------|----------|
| `example.com/wiki/Aviation` | `example.com/index.php?topic=42&area=3a5ebc944f41daa6f849f730f1` |
| `example.com/summer-clothing` | `example.com/summer_clothing`（用连字符，不用下划线） |
| 使用受众语言 | 在非英文网站用英文 URL |
| 尽量少用参数 | 带大量追踪参数 |

### 具体规则

1. **用连字符 `-` 分隔字词**，不用下划线 `_`
2. **非 ASCII 字符需百分比编码**：`%E6%9D%82%E8%B4%A7` 优于 `杂货`
3. **注意大小写**：Google 将 `/APPLE` 和 `/apple` 视为不同网址
4. **减少参数**：删除不改变内容的参数
5. **描述性网址**：网址中包含对用户有用的字词

### 复杂 URL 的危害

过于复杂的 URL 会导致：
- Googlebot 消耗大量不必要的带宽
- Google 搜索可能无法将所有内容完整编入索引
- 常见问题：累加过滤、不相关参数、会话 ID、动态日历、损坏的相对链接

## 四、图片优化

### 核心原则

| 要素 | 说明 |
|------|------|
| **高画质** | 使用清晰可辨的高质量图片 |
| **上下文关联** | 将图片放在相关文字附近 |
| **描述性 Alt 文本** | 帮助搜索引擎理解图片内容和与页面的关系 |

### 图片 Alt 文本

Alt 文本是简短的描述性文本，说明图片与内容的关系：

```html
<!-- ✅ 良好 -->
<img src="yarn-shop.jpg" alt="伦敦纱线店的毛线展示架">

<!-- ❌ 欠佳 -->
<img src="yarn-shop.jpg" alt="图片">
<img src="yarn-shop.jpg" alt="">
```

### Article 结构化数据中的图片要求

- 每个网页**必须包含至少1张图片**
- 图片网址必须**可抓取且可编入索引**
- 图片必须**代表标记的内容**（不能用 logo）
- 建议提供**多个高分辨率图片**（宽×高至少 50K 像素）
- 推荐宽高比：**16:9、4:3、1:1**

## 五、Article 结构化数据 (JSON-LD)

### 推荐属性

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "NewsArticle",
  "headline": "文章标题",
  "image": [
    "https://example.com/photos/1x1/photo.jpg",
    "https://example.com/photos/4x3/photo.jpg",
    "https://example.com/photos/16x9/photo.jpg"
  ],
  "datePublished": "2026-04-27T08:00:00+08:00",
  "dateModified": "2026-04-27T09:20:00+08:00",
  "author": [{
    "@type": "Person",
    "name": "作者名",
    "url": "https://example.com/profile/author"
  }]
}
</script>
```

### 作者标记最佳实践

| 规则 | 正确 ✅ | 错误 ❌ |
|------|---------|---------|
| 添加所有作者 | 每位作者单独字段 | 合并为 "A, B" |
| author.name 仅写名字 | "张三" | "资深编辑张三" |
| 使用 type 和 url | Person + url 链接 | 不提供标识链接 |
| 正确 type | 个人用 Person | 对人用 Organization |

### 实施流程

1. 添加尽可能多的建议属性
2. 使用**富媒体搜索结果测试**验证
3. 部署后用**网址检查工具**确认可访问
4. 提交**站点地图**通知 Google

## 六、链接优化

### 锚文本原则

| 应该 ✅ | 不应该 ❌ |
|---------|----------|
| 描述清楚、措辞简洁 | "点击此处"、"了解更多" |
| 具有相关性、设定预期 | 关键字堆砌 |
| 注重上下文 | 锚文本过长（整个句子/段落） |
| 链接间有适当间距 | 链接彼此紧紧相连 |

### 内链策略

- 每个重要网页**至少有 1 个其他页面**链接到它
- 在相关上下文中提供指向相关资源的链接
- **没有神奇的理想链接数量**，但"感觉太多就是太多"

### 技术要求

- 链接必须是**含 `href` 属性的 `<a>` 元素**
- Google 无法从 `onclick`、`routerLink` 等中提取网址
- 图片链接使用 `alt` 属性作为锚文本
