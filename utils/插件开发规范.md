# 插件开发规范

> 本规范定义了 `utils/` 目录下插件模块的代码结构、命名约定、日志控制、Django 集成等规则。
> 所有新插件必须遵守本规范，确保风格统一、可维护、可复用。

---

## 1. 目录结构

```
utils/
├── 模块文件数据库/          # 插件源码（纯 Python，不依赖 Django）
│   ├── sitemap_generator.py
│   ├── image_watermark.py
│   └── your_new_plugin.py   # ← 新插件放这里
│
├── 模块文档/                # 插件使用教程（Markdown）
│   ├── 站点地图生成器使用教程.md
│   ├── 图片水印工具使用教程.md
│   └── 你的新插件使用教程.md # ← 新文档放这里
│
└── 插件开发规范.md          # ← 你正在读的这个文件
```

**规则**：
- 源码文件和文档文件**一一对应**，名称互相关联
- 源码文件名用 **snake_case**（如 `sitemap_generator.py`）
- 文档文件名用**中文描述 + "使用教程"**（如 `站点地图生成器使用教程.md`）

## 2. 源码文件规范

### 2.1 文件模板

每个插件源码文件必须包含以下结构：

```python
"""
插件中文名 - 一句话描述

功能：
  1. 核心功能点 1
  2. 核心功能点 2
  3. 核心功能点 3

环境变量（.env）：
  ENV_VAR_NAME     - 说明，默认值
  ENV_VAR_NAME_2   - 说明，默认值
  DEBUG            - 调试日志开关，True 时输出 loguru 日志

使用示例：
  # Python 调用
  from plugin_module import PluginClass
  obj = PluginClass()
  result = obj.core_method(...)

  # 命令行
  python plugin_module.py --arg1 value1 --arg2 value2
"""

import os
import sys
import json
import argparse
from typing import List, Dict, Optional

from loguru import logger

# ============================================================
# 日志控制：仅在 DEBUG=True 时输出
# ============================================================
DEBUG = os.getenv("DEBUG", "False").lower() in ("true", "1", "yes")
if not DEBUG:
    logger.disable("__main__")


class PluginClass:
    """插件主类 - 一句话描述

    核心用法：
        obj = PluginClass()
        result = obj.core_method(...)

    Attributes:
        attr1: 属性说明
        attr2: 属性说明
    """

    def __init__(self, ...):
        """初始化

        Args:
            param1: 参数说明

        Raises:
            ValueError: 参数校验失败时
        """
        # 校验必填参数
        if not param1:
            raise ValueError("param1 不能为空")

        # 从 .env 读取默认配置（允许运行时覆盖）
        self.attr1 = param1 or os.getenv("ENV_VAR_NAME", "默认值")

        logger.info(f"插件初始化: attr1={self.attr1}")

    def core_method(self, ...) -> str:
        """核心方法 - 一句话描述

        Args:
            param: 参数说明

        Returns:
            返回值说明

        Raises:
            ValueError: 参数校验失败时
        """
        logger.info(f"开始执行: ...")
        # 业务逻辑
        logger.info(f"执行完成")
        return result

    # ----------------------------------------------------------
    # 内部方法（以下划线开头）
    # ----------------------------------------------------------

    def _helper_method(self, ...):
        """内部辅助方法"""
        ...


# ============================================================
# CLI 命令行接口
# ============================================================

def main():
    """命令行入口"""
    parser = argparse.ArgumentParser(
        description="插件中文名 - 一句话描述",
        formatter_class=argparse.RawDescriptionHelpFormatter,
        epilog="""
使用示例：
  python plugin_module.py --arg1 value1
  python plugin_module.py --arg1 value1 --output result.txt
        """,
    )

    parser.add_argument("--arg1", required=True, help="参数说明")
    parser.add_argument("--output", "-o", help="输出文件路径（不指定则输出到 stdout）")

    args = parser.parse_args()

    try:
        obj = PluginClass(...)
        result = obj.core_method(...)
    except ValueError as e:
        print(f"错误: {e}", file=sys.stderr)
        sys.exit(1)

    if args.output:
        with open(args.output, "w", encoding="utf-8") as f:
            f.write(result)
        print(f"已保存: {args.output}")
    else:
        print(result)


if __name__ == "__main__":
    main()
```

### 2.2 必须遵守的规则

| # | 规则 | 说明 |
|---|------|------|
| 1 | **模块顶部 docstring** | 包含：功能、环境变量、使用示例，缺一不可 |
| 2 | **类为主体** | 插件逻辑封装在一个主类中，不要散落为多个顶层函数 |
| 3 | **type hints** | 所有公开方法的参数和返回值必须有类型注解 |
| 4 | **完整 docstring** | 公开方法的 docstring 必须包含 Args / Returns / Raises |
| 5 | **loguru 日志** | 禁止使用 `print()`，统一用 `logger.info/debug/warning/error` |
| 6 | **DEBUG 控制** | 日志必须在 `DEBUG=False` 时静默（见 2.1 模板） |
| 7 | **CLI 接口** | 必须提供 `argparse` 命令行入口和 `if __name__ == "__main__"` |
| 8 | **参数校验** | `__init__` 和核心方法必须校验必填参数，失败抛 `ValueError` |
| 9 | **环境变量读取** | 配置从 `.env` 读取，但允许运行时参数覆盖（`param or os.getenv(...)`） |
| 10 | **不依赖 Django** | 源码文件必须是纯 Python，不得 `import django` 或引用项目模块 |

### 2.3 日志控制模板

每个插件文件顶部必须有这段代码，**位置固定、格式固定**：

```python
from loguru import logger

# ============================================================
# 日志控制：仅在 DEBUG=True 时输出
# ============================================================
DEBUG = os.getenv("DEBUG", "False").lower() in ("true", "1", "yes")
if not DEBUG:
    logger.disable("__main__")
```

## 4. Django 集成规范

插件源码是纯 Python，但通常需要通过 Django 视图对外提供服务。集成模式如下：

### 4.1 配置层

在 `.env` 中添加插件所需的环境变量，然后在 `WebModel/settings.py` 中读取：

```python
# .env
NEW_PLUGIN_CONFIG=some_value

# settings.py
NEW_PLUGIN_CONFIG = os.getenv('NEW_PLUGIN_CONFIG', 'default_value')
```

**规则**：
- 环境变量名用**大写 + 下划线**，如 `SITE_SLUG`、`ARTICLE_IMAGE_WATERMARK_TEXT`
- `settings.py` 中的变量名与 `.env` 保持一致

### 4.2 视图层

在 `DZY_Web/views/request.py` 中添加视图函数：

```python
# 加载插件模块（通过 sys.path.insert 导入）
import sys
sys.path.insert(0, os.path.join(settings.BASE_DIR, 'utils', '模块文件数据库'))
from plugin_module import PluginClass


def site_plugin_view(request):
    """插件功能 - 一句话描述

    访问 /web/{SITE_SLUG}/plugin-path/ 即可使用。
    """
    site_slug = _get_site_slug()
    
    # 读取配置
    config = getattr(settings, 'NEW_PLUGIN_CONFIG', 'default')
    
    # 调用插件核心逻辑
    obj = PluginClass(config=config)
    result = obj.core_method(...)
    
    # 返回响应（根据场景选择 HttpResponse / render / JsonResponse）
    return HttpResponse(result, content_type='...')
```

**规则**：
- 视图函数名以 `site_` 开头，如 `site_sitemap`、`site_rss`
- 使用 `sys.path.insert` 导入插件模块，不用 `from utils.模块文件数据库.xxx`
- 配置从 `settings` 读取，不直接 `os.getenv()`
- 使用已有的辅助函数：`_get_site_slug()`、`_get_site_articles()`

### 4.3 路由层

在 `DZY_Web/views/urls.py` 中注册路由：

```python
urlpatterns = [
    # ... 已有路由 ...
    path(f'{site_slug}/plugin-path/', request.site_plugin_view, name='site_plugin_name'),
]
```

**规则**：
- 路由前缀使用 `{site_slug}`，不硬编码站点名
- URL name 使用 `site_` 前缀，如 `site_sitemap`
- 路由路径符合 SEO 和 RESTful 习惯（如 `sitemap.xml`、`rss.xml`）

### 4.4 上下文注入（可选）

如果插件配置需要在模板中使用，在 `DZY_Web/context_processors.py` 的 `site_context` 中添加：

```python
def site_context(request):
    slug = settings.SITE_SLUG
    return {
        # ... 已有变量 ...
        'plugin_config': getattr(settings, 'NEW_PLUGIN_CONFIG', ''),
    }
```

## 5. 环境变量命名规范

| 类型 | 前缀 | 示例 |
|------|------|------|
| 站点级配置 | `SITE_` | `SITE_SLUG`、`SITE_NAME` |
| 文章相关 | `ARTICLE_` | `ARTICLE_IMAGE_WATERMARK_TEXT` |
| 插件专属 | `插件名_` | `SITEMAP_XXX`（如需要） |
| 全局开关 | 无前缀 | `DEBUG`、`SECRET_KEY` |

---

## 6. 禁止事项

| # | 禁止 | 原因 |
|---|------|------|
| 1 | 在插件源码中 `import django` | 插件必须可独立运行和测试 |
| 2 | 硬编码域名、IP、端口 | 必须从 `.env` 的 `ALLOWED_HOSTS` 读取，`settings.py` 自动推导 `SITE_URL` |
| 3 | 硬编码站点路径（如 `youdaotools`） | 必须从 `.env` 的 `SITE_SLUG` 读取 |
| 4 | 使用 `print()` 输出日志 | 统一用 `loguru`，由 `DEBUG` 控制 |
| 5 | 插件间互相依赖 | 每个插件必须可独立使用 |
| 6 | 在源码中写项目绝对路径 | 使用参数传入或 `os.path.join(BASE_DIR, ...)` |
| 7 | 省略 CLI 接口 | 每个插件必须可通过命令行独立执行 |

## 7. 开发检查清单

新插件开发完成后，逐项检查：

- [ ] 源码文件放在 `utils/模块文件数据库/`
- [ ] 文档文件放在 `utils/模块文档/`，名称为「插件中文名 + 使用教程」
- [ ] 源码顶部 docstring 包含：功能、环境变量、使用示例
- [ ] 主类有完整 docstring（Attributes、核心方法有 Args/Returns/Raises）
- [ ] 所有公开方法有 type hints
- [ ] 日志使用 loguru，DEBUG=False 时静默
- [ ] 提供 CLI 接口（argparse + `if __name__ == "__main__"`）
- [ ] 不依赖 Django，可独立运行
- [ ] 无硬编码域名、端口、站点路径
- [ ] 配置从 `.env` 读取，运行时可覆盖
- [ ] 如需 Django 视图，已在 `request.py` / `urls.py` 中注册
- [ ] 如有新环境变量，已在 `.env` 和 `settings.py` 中添加
- [ ] 错误处理完善（必填校验 + 异常抛出 + 降级策略）
